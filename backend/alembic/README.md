Alembic Migration Guide
═══════════════════════════════════════════════════════════════

SETUP (first time only)
───────────────────────
Alembic is already initialized. Do not run alembic init again.

COMMON COMMANDS
───────────────────────

Apply all pending migrations:
  alembic upgrade head

Rollback one migration:
  alembic downgrade -1

Rollback to a specific revision:
  alembic downgrade <revision_id>

Rollback everything (nuclear):
  alembic downgrade base

Check current applied revision:
  alembic current

Show full migration history:
  alembic history --verbose

Auto-generate a new migration after model changes:
  alembic revision --autogenerate -m "describe your change here"

Create a blank migration (for manual SQL):
  alembic revision -m "describe your change here"

Preview SQL without applying (offline mode):
  alembic upgrade head --sql

WORKFLOW FOR NEW MODEL CHANGES
───────────────────────────────
1. Edit your SQLAlchemy model in app/models/
2. Import the model in alembic/env.py if it is new
3. Run: alembic revision --autogenerate -m "add column X to table Y"
4. Open the generated file in alembic/versions/ and REVIEW it
5. Run: alembic upgrade head
6. Verify with: alembic current

IMPORTANT RULES
───────────────────────────────
- Never edit a migration file that has already been applied to production
- Always review autogenerated migrations before running them
- Keep migrations small and focused — one change per migration
- Always write a working downgrade() function
- Never delete migration files from versions/

NAMING CONVENTION
───────────────────────────────
Files are named: YYYY_MM_DD_HHMMSS_<rev>_<slug>.py
Example: 2024_01_15_143022_0002_add_category_to_products.py
